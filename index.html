This specification defines a web manifest, which is a JSON document that a user agent can use to provide enhanced bookmarking capabilities. Unlike regular bookmarks, this specification allows developers to specify a range of icons, the name of the web application, as well as the url to launch when a website is bookmarked to, for example, the homescreen of a device. It also defines a special "standalone" mode in which applications can be launched, which is intended to allow web applications to run independently of a web browser and more tightly integrate with native application environments.

This specification also defines two ways to allows a web page to be associated with a web manifest: through the manifest link type or through bookmark API.

Implementors should be aware that this specification is not stable. Implementors who are not taking part in the discussions are likely to find the specification changing out from under them in incompatible ways. Vendors interested in implementing this specification before it eventually reaches the Candidate Recommendation phase should join the aforementioned mailing lists and take part in the discussions.

There is only one class of product that can claim conformance to this specification: a user agent.

Web manifest

A web manifest is a [[!JSON]] document that consists of a top-level object (the web manifest root) and one or more members, some of which contain other objects.

name member

The name is a string that represents the name of the application as is usually displayed to the user (e.g., amongst a list of other applications, or as a label underneath an icon).

The steps for processing the name member is given by the following algorithm. The algorithm takes a web manifest as an argument and returns either an error or a string:

  1. Let value be the result of calling the [[\GetOwnProperty]] internal method of manifest passing "name" as the argument.
  2. If value is null or undefined, then let value be an implementation specific string that can serve as a suitable name for the application. For example, if the manifest was obtained from a link element and the document has a title, then the document's title could be used. Alternatively, the user could be prompted to provide a custom name.
  3. Let result be the result of calling ToString(value)
  4. Strip leading and trailing whitespace from result.
  5. Return the result.

altname member

The altname is a string that represents an alternative name of the application, such as the name under which the application is sold or marketed.

The steps for processing the short name member is given by the following algorithm. The algorithm takes a web manifest as an argument and returns a string:

  1. Let value be the result of calling the [[\GetOwnProperty]] internal method of manifest passing "altname" as the argument.
  2. If value is null or undefined, then let value be an implementation specific string that can serve as a suitable alt for the application.
  3. Let result be the result of calling ToString(value).
  4. Strip leading and trailing whitespace from result.
  5. Return the result.

url member

The url member is the URL that is is loaded when the application is launched. When it's missing from the manifest, the UA loads the URL from the manifest was loaded.

The steps for processing the url member are given by the following algorithm. The algorithm returns a URL.

  1. Let value be the result of calling the [[\GetOwnProperty]] internal method of manifest passing "url" as the argument.
  2. If value is null or undefined or Type(value) is not "string":
    1. Let result be the absolute URL from which the manifest was fetched from.
  3. Otherwise:
    1. Let manifest URL be the URL from which the manifest was fetched from.
    2. Parse value, using manifest URL as the base URL, and let result be the result.
  4. Return result.

icons member

The icons is a list of icon objects that represents a set of icons that the application can make use.

The steps for processing the icons member are given by the following algorithm. The algorithm returns either an list of icons, which can be empty.

  1. Let icons be an empty list.
  2. Let manifest URL be the URL from which the manifest was fetched from.
  3. Let value be the result of calling the [[\GetOwnProperty]] internal method of manifest passing "icons" as the argument.
  4. If value is an array, then for each potential icon in the array:
    1. Let src be the result of running the steps for processing the src member of an icon. If the result is an error, stop processing this potential icon and move to the next portential icon in value (if available).
    2. Run the steps for processing the type member of an icon. If the result is an error,
    3. Let width be the result of running the steps for processing the width member of an icon. If the result is an error,
    4. Otherwise, let icon be an object.
  5. return icons.

orientation member

The orientation member represents the screen orientation required to use the web application effectively. Orientation is only applicable for applications whose mode of operation is standalone.

A screen orientation is a string that conforms to the following ABNF. 

  orientation = type ["-" option]
  type = "portrait" / "landscape"
  option = "locked" / "primary" / "secondary"

The steps for processing the orientation member are given by the following algorithm. The algorithm returns a string.

Icon object and its members

Each icons object represents an icon for an application suitable to use at some dimensions and screen density.

width and height members

The width and height members represent the natural width of the icon in pixels. Their corresponding value is a positive number greater than 0.

density member

The density member of an icon is the device pixel density for which this icon was designed. The device pixel density is expressed as the number of dots per 'px' unit (equivalent to a dppx as defined in [[!CSS-VALUES]]). The value is a positive number greater than 0. When the value is missing, the user agent assumes the value 1.

src member

The src member of an icon is a URL from which the icon can be fetched.

The steps for processing the src member of an icon are given by the following algorithm. The algorithm takes a potential icon object as an arugment. The algorithm returns a URL or an error.

  1. Let raw src be the result of calling the [[\GetOwnProperty]] internal method of potential icon passing "src" as the argument.
  2. If raw src is null or underfined or Type(potential src) is not "string", return an error.
  3. Parse raw src, using manifest URL as the base URL, and let src be the result.
  4. return result.

type member

The type member of an icon is a hint as to the media type of the icon. A user agent can ignore icons of media types it does not support.

Processing the web manifest

Algorithms in this specification use the conventions described in [[!ECMA-262]], such as the use of steps and sub-steps, and so on. The ToNumber, ToString abstract operations and the Type(x) notation referenced in this section are defined in [[!ECMA-262]]. Processing also relies on various algorithms defined in [[!HTML]], [[!FETCH]], and [[!URL]].

As the web manifest format is a [[!JSON]] document, this specification relies on the types defined in [[!JSON]] specification: namely object, array, number, and string. Strict type checking is not enforced by this specification. Instead, each member's definition specifies the steps required to process a particular member.

In the algorithms, to ignore means that the user agent MUST act as if the developer had not declared the particular member in the web manifest document.

Certain error conditions can result in a web manifest being treated as an invalid web manifest. An invalid web manifest is a web manifest that is deemed to be non-conforming in such a way that it would not be possible for the user agent to continue processing (e.g., it can't be parsed by the JSON parser because of a syntax error). In such a case, the user agent MAY report the error to the user in a user-agent-specific manner, MAY navigate to an error page to report the error, or MAY ignore the error and do nothing. In either case, when a step results in an invalid web manifest the user agent MUST abort whatever step or sub-step caused the condition.

The steps for processing a web manifest are given by the following algorithm:

  1. Let text be the result of fetching the web manifest. If the fetching algorithm aborts, treat this web manifest as an invalid web manifest.
  2. Let manifest be the result of invoking the parse function of the JSON object defined in [[!ECMASCRIPT]] with text as its only argument. If parsing throws an exception, then treat this web manifest as an invalid application manifest.
  3. Let name be the result of running the steps for processing the name member.
  4. Let alternative name be the result of running the steps for processing the altname member.
  5. Let mode be the result of running the steps for processing the mode member.
  6. Let orientation be the result of running the steps for processing the orientation member.
  7. Let url be the result of running the steps for processing the url member.
  8. Let icons be the result of running the steps for processing the icons member.
  9. Optionally, ignore all other members.

Common processing infrastructure

The steps for processing a member whose type is a positive number are given by the following algorithm. The algorithm takes an object and a key as an argument. The algorithm returns a positive number, undefined, or an error.

  1. Let value be the result of calling the [[\GetOwnProperty]] internal method of object passing key as the argument.
  2. If value is undefined, return undefined.
  3. Let number be the result of parseFloat(value);
  4. If number is NaN, +0, −0, +∞, or −∞, or less than 0, return an error.
  5. Return number.

Linking to a web manifest

The manifest keyword can be used with a [[!HTML]] link element. This keyword creates an external resource link.

Link type Effect on... Brief description
link a and area
manifest External Resource not allowed Imports or links to a web manifest.

The default media type for resources associated with the manifest link type is application/webapp-manifest+json.

In cases where more than one link element with a manifest link type appears in a document, the user agent MUST use the first inserted link element and ignore all subsequent link elements with a manifest link type (even if the first element was in error).

The appropriate time to obtain the web manifest is when the external resource link is created or when its element is inserted into a document, whichever happens last. However, a user agent MAY opt to delay obtaining a web manifest until after the document and its other resources have loaded (i.e., to not delay the availability of content and scripts required by the document).

Extending the manifest

Although proprietary extensions are undesirable, they can't realistically be avoided. As such, the RECOMMENDED way to add proprietary extension is to use a vendor prefix.

Security considerations:

As the web manifest format is JSON and will commonly be encoded using [[!UNICODE]], the security considerations described in [[!JSON]] and [[!UNICODE-SECURITY]] apply. In addition, implementors need to impose their own implementation-specific limits on the values of otherwise unconstrained member types, e.g. to prevent denial of service attacks, to guard against running out of memory, or to work around platform-specific limitations.

The web manifest document allows developeres, through the permissions and permissions members, to request permission to enable security sensitive APIs. As these APIs are outside the scope of this specification, significant caution needs to be taken when granting an application the capability to use a feature. The relevant APIs define their own security considerations (see the System Applications Working Group's APIs).

Web applications will generally contain ECMAScript, HTML, CSS files, and other media, which are executed in a sand-boxed environment. As such, implementors need to be aware of the security implications for the types they support. Specifically, implementors need to consider the security implications outlined in the [[!CSS-MIME]] specification, the [[!ECMAScript-MIME]] specification, and the [[!HTML]] specification.

As web applications can contain content that is able to simultaneously interact with the local device and a remote host, implementors need to consider the privacy implications resulting from exposing private information to a remote host. Mitigation and in-depth defensive measures are an implementation responsibility and not prescribed by this specification. However, in designing these measures, implementors are advised to enable user awareness of information sharing, and to provide easy access to interfaces that enable revocation of permissions.

As this specification allows for the declaration of IRIs within certain members of the web manifest, implementors need to consider the security considerations discussed in the [IRI] specification. Implementations intending to display IRIs and IDNA addresses found in the web manifest are strongly encouraged to follow the security advice given in [[!UNICODE-SECURITY]].
